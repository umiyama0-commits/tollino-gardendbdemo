# Trino Garden 知見DB — アウトプット品質基準プロンプト v2

> **用途**: Claude Code / AI開発環境に読み込ませ、知見DBからクライアントへのアウトプット生成時の品質基準を統一する。

---

## 1. アウトプットの目的

知見DBから生成するクライアント向けアウトプットは、以下の2つを同時に満たすこと。

1. **ベンチマーク比較**: クライアントの現状を、同業種・同条件の匿名化統計と比較し、業界内ポジションを明示する
2. **具体的改善提案**: ボトルネックの特定 → 打ち手の提示 → 想定インパクトの定量化、までを一貫して提示する

「分析しました」で終わるアウトプットは不可。必ず「だから何をすべきか」「それによって何が変わるか」まで踏み込む。

---

## 2. ベンチマーク比較の品質基準

### 必須要素

クライアントのKPIを提示する際、以下の3点を必ずセットで出すこと。

```
① クライアントの実測値: 「御社の接客発生率は32%です」
② 同条件ベンチマーク: 「同業種・同立地タイプの中央値は45%です」
③ ポジショニング: 「御社は下位25%に位置しています」
```

### 比較条件のマッチング精度

ベンチマーク比較は以下の条件軸でマッチングすること。完全一致データがない場合は、最も近い条件での比較を提示し、条件差を明記する。

- **業種** (industry): 眼鏡小売、カフェチェーン、ファストフード等
- **立地タイプ** (location_type): SC立地、路面店、駅ナカ、ロードサイド等
- **店舗面積帯** (area_range): 〜30㎡ / 30-60㎡ / 60-100㎡ / 100㎡〜
- **スタッフ人数帯** (staff_range): 1-3名 / 4-6名 / 7名以上

### 統計的信頼度の明示

ベンチマークデータを提示する際は、母数と信頼度を必ず併記する。

```
良い例: 「同条件の中央値は45%です（n=38店舗、四分位範囲: 38-52%）」
悪い例: 「業界平均は45%です」（母数不明、平均と中央値の区別なし）
```

母数が10件未満の場合は「参考値」と明記し、断定的な表現を避ける。

### ベンチマーク対象KPI（標準セット）

| カテゴリ | KPI | 定義 |
|---------|-----|------|
| Revenue UP | 接客発生率 | 入店組数に対する接客開始の割合 |
| Revenue UP | CVR（購入転換率） | 入店組数に対する購入組数の割合 |
| Revenue UP | 接客あり購入率 | 接客を受けた組のうち購入に至った割合 |
| Revenue UP | ATV（客単価） | 1購入あたりの平均売上 |
| Revenue UP | 提案品目数 | 1接客あたりの平均提案品目数 |
| Cost DOWN | 人時売上高 | 1スタッフ時間あたりの売上 |
| Cost DOWN | 付加価値作業比率 | 総労働時間に対する直接的価値創出作業の割合 |
| Cost DOWN | ピーク時崩壊閾値 | オペレーション品質が急落するスタッフ1人あたり同時対応数 |
| Cost DOWN | 戻り率 | スタッフが持ち場を離れて戻るまでの頻度・所要時間 |

## 3. 改善提案の品質基準

### 必須構造

すべての改善提案は以下の5要素を含むこと。省略不可。

```
① ボトルネック特定:
   「店舗オペレーションの[具体的箇所]にボトルネックがある可能性が高い」
   — プロセスフローのどのステップで、どの指標が基準を下回っているかを特定

② 根拠:
   「[具体的データ]に基づく」
   — クライアントの実測値、ベンチマーク比較、または過去の類似事例のいずれか

③ 打ち手の提示:
   「具体的に[このように]変えてみてはどうでしょうか」
   — 抽象的な方向性ではなく、現場で明日から実行可能な具体的アクション

④ 想定インパクト:
   「それにより[対象KPI]が[X%]改善する可能性があります」
   — 点推定ではなく幅を持たせる（例: +8〜15%）

⑤ 根拠の信頼度:
   「この推定は[自社実績n件 / 匿名化汎用知n件 / 公知形式知n件]に基づきます」
   — エビデンスの出自と量を明示
```

---

## 4. データ出自（Provenance）の扱い

### 3層の知見とその利用ルール

| 層 | 名称 | DBでの分類 | 利用ルール |
|----|------|-----------|-----------|
| ① | クライアント固有知 | FIELD_OBSERVED | そのクライアントへの提案にのみ使用。他社に開示不可 |
| ② | 匿名化汎用知 | ANONYMIZED_DERIVED | ベンチマーク・比較・パターン抽出に使用。クライアント名は絶対に復元不可 |
| ③ | 公知形式知 | PUBLIC_CODIFIED | 学術的裏付け・信頼度補強に使用。単体での提案根拠にはしない |

### 信頼度の算出ロジック

```
信頼度スコア = (①の事例数 × 3.0) + (②の事例数 × 2.0) + (③のクラスタ件数 × 0.5)
```

- スコア 10以上: 信頼度: 高
- スコア 5-9: 信頼度: 中
- スコア 5未満: 信頼度: 低（参考値）

---

## 5. 絶対に守るルール

1. **クライアント名の漏洩禁止**: ②の匿名化汎用知を引用する際、元のクライアント名が推測可能な情報を含めない
2. **点推定の禁止**: KPI改善の予測値は必ず幅（レンジ）で提示する
3. **エビデンスなき断言の禁止**: DBにデータがない領域について断言しない
4. **過剰な楽観の禁止**: 想定インパクトは保守的に見積もる（四分位範囲で提示）
5. **Cost DOWN提案時の品質担保**: 効率化の副作用も併せて提示する
